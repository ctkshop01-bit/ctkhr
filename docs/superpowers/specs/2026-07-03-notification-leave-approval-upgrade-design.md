# 通知中心、假期结转与审批升级设计

日期：2026-07-03

## 1. 背景

当前系统已经具备以下能力：

- 员工端：打卡、请假/加班申请、公告查看、任务查看、工资查看
- 管理端：员工管理、审批中心、公告发布、任务创建、工资生成、数据看板
- 数据架构：单主机共享 JSON 数据库，前后端通过共享快照与命令接口读写

当前缺口主要有 3 个：

- 缺少独立的站内消息中心，审批结果、公告、任务和异常处理结果无法形成统一消息流
- 假期规则仍以静态月度额度为主，缺少假期余额账本和可配置的月度结转能力
- 审批中心虽已支持请假、加班、异常处理，但缺少审批人配置与统一审批日志，审计链不完整

本次目标是在不推翻现有共享数据架构的前提下，以叠加方式完成：

- 消息提醒中心
- 假期额度与自动结转
- 审批流升级

## 2. 目标

### 2.1 业务目标

- 让员工能在一个地方看到与自己相关的审批结果、公告和任务提醒
- 让假期从“静态规则”升级为“可追踪余额账本 + 月度结转规则”
- 让审批链具备明确审批人配置与统一审计记录

### 2.2 工程目标

- 继续复用现有 `dbStore -> sharedApi -> app-db-commands.mjs` 模式
- 所有新结构都兼容历史数据，缺省为空时系统仍可正常运行
- 不引入外部消息依赖，不接短信、邮件、微信等站外通道
- 不一次性重写历史库，采用懒生成与增量补强策略

## 3. 范围

### 3.1 本次包含

- 员工端新增站内消息中心
- 管理端与员工端增加未读消息计数入口
- 新增消息数据结构与已读状态
- 新增假期余额账本
- 新增假期结转规则配置
- 工资生成逻辑改为基于余额账本核算带薪/无薪假
- 新增审批人配置
- 新增统一审批日志
- 审批结果自动生成站内消息

### 3.2 本次不包含

- 短信、邮件、LINE、微信等站外提醒
- 一审二审、会签、抄送等多级审批
- 完整 BPM 审批引擎重构
- 历史全库一次性迁移重算
- 任务按人精细分派重构

## 4. 总体方案

本次采用“轻量叠加式”方案：

- 继续沿用现有页面与命令体系
- 在现有数据模型旁新增 4 类结构：
  - `notifications`
  - `leaveBalances`
  - `approvalSettings`
  - `approvalLogs`
- 在现有页面中追加入口与只读信息，不大幅推翻原页面布局
- 在共享后端命令层统一处理新逻辑，前端 store 只负责调用命令和刷新快照

该方案优点：

- 风险低，适合连续上线
- 与当前共享数据版完全兼容
- 能把三项功能接进现有员工端/管理端主链路

## 5. 数据模型

### 5.1 notifications

用于员工站内消息中心。

字段建议：

- `id`
- `userId`
- `type`
- `title`
- `body`
- `relatedType`
- `relatedId`
- `isRead`
- `createdAtISO`

`type` 首版固定为：

- `leave_reviewed`
- `overtime_reviewed`
- `attendance_confirmed`
- `announcement_published`
- `task_created`

说明：

- 所有消息面向员工，不单独建设管理员消息中心
- 公告和任务消息可面向所有 `active` 员工批量生成
- 历史无此字段时默认为空数组

### 5.2 leaveBalances

用于月度假期额度、结转、实际使用与结余管理。

字段建议：

- `userId`
- `month`
- `grantedDays`
- `carriedDays`
- `usedPaidDays`
- `usedUnpaidDays`
- `expiredDays`
- `closingBalanceDays`

说明：

- 账本按“员工 + 月份”维护
- 余额不直接写回 `users`，避免后续难以追踪
- 历史月份无账本时采用懒生成

### 5.3 leavePolicySettings

用于控制月度发放与结转规则。

字段建议：

- `carryoverMode`
- `carryoverCapDays`
- `monthlyGrantMode`

规则枚举：

- `carryoverMode = none | full | capped`
- `monthlyGrantMode` 首版可以固定为按员工字段发放，但保留配置位方便扩展

### 5.4 approvalSettings

用于配置事项审批人。

字段建议：

- `leaveReviewerUserId`
- `overtimeReviewerUserId`
- `attendanceReviewerUserId`

说明：

- 首版每个事项仅允许一个当前审批人
- 若审批人停用或不存在，视为未配置

### 5.5 approvalLogs

用于统一审计审批动作。

字段建议：

- `id`
- `requestType`
- `requestId`
- `submitterUserId`
- `reviewerUserId`
- `decision`
- `note`
- `createdAtISO`

说明：

- 请假、加班、异常考勤都写统一审批日志
- 现有请假/加班记录中的 `reviewerId / reviewNote / reviewedAtISO` 保留不删
- 审批日志是补强，不替换原数据结构

## 6. 页面与交互

### 6.1 员工端

#### 消息中心

新增消息页，提供：

- 全部消息列表
- 未读筛选
- 全部已读
- 点击消息后跳转到相关页面并标记已读

入口建议：

- 桌面侧边栏增加“消息中心”
- 移动端底部导航或顶部角标增加未读提示
- 工作台可补一个“未读消息”摘要区

#### 工作台

新增：

- 未读消息数角标或摘要卡片
- 假期余额摘要，如“本月可用带薪假 X 天”

#### 请假申请页

新增只读提示：

- 当前月可用带薪假
- 本次申请预计消耗带薪/无薪天数

### 6.2 管理端

#### 审批设置页

新增独立页面用于配置：

- 请假审批人
- 加班审批人
- 异常考勤审批人

若未配置审批人：

- 后台高亮提示
- 对应事项不允许员工提交

#### 审批中心

增强展示：

- 当前事项审批人
- 审批操作写入统一日志
- 审批详情中可显示最近一次审批记录摘要

#### 员工管理页

新增只读区：

- 当前假期余额
- 当月发放与结转摘要

#### 工资页

保留现有工资展示结构，但核算来源改为：

- 当月账本中的 `usedPaidDays`
- 当月账本中的 `usedUnpaidDays`
- 对应无薪假扣款结果

## 7. 业务规则

### 7.1 消息生成规则

- 请假通过/驳回：给申请员工生成 1 条消息
- 加班通过/驳回：给申请员工生成 1 条消息
- 异常考勤确认/修正：给对应员工生成 1 条消息
- 管理员发布公告：给所有 `active` 员工生成 1 条消息
- 管理员创建任务：按当前任务可见范围发消息；若当前为全员任务，则发给所有 `active` 员工
- 员工点击消息后标记已读
- 员工登录和页面切换时，未读数随共享快照刷新

### 7.2 假期额度与结转规则

- 每月先发放当月带薪假额度
- 默认额度来源仍是员工字段 `monthlyPaidLeaveDays`
- `regular` 一般有额度，`probation` 可为 `0`
- 当月请假先消耗可用带薪余额
- 超出可用带薪余额的部分记为无薪假
- 月底根据配置执行结转：
  - `none`：清零
  - `full`：全额结转
  - `capped`：按上限结转，超出失效
- 历史月份无账本时，在首次访问或首次工资生成时懒生成

### 7.3 审批规则

- 每个事项仅一个审批人
- 请假、加班、异常考勤分别独立配置审批人
- 未配置审批人时，对应事项禁止员工提交，并给出明确提示
- 审批动作必须写入 `approvalLogs`
- 请假/加班原记录继续写 `reviewerId / reviewNote / reviewedAtISO`
- 异常考勤确认也写统一审批日志
- 审批人停用或删除时，该事项自动回到“未配置审批人”状态

## 8. 服务端命令与数据流

### 8.1 需要新增或扩展的命令

- 读取消息列表
- 标记单条消息已读
- 标记全部消息已读
- 读取/刷新假期余额账本
- 更新假期结转规则
- 更新审批人配置
- 读取审批日志
- 在现有 `submitLeave / reviewLeave / submitOvertime / reviewOvertime / confirmAttendance / createAnnouncement / createTask / generatePayroll` 中补消息与账本联动

### 8.2 数据流

员工提交请假：

1. 前端提交请假命令
2. 服务端先校验该事项审批人是否已配置
3. 若未配置则拒绝写入并返回明确错误
4. 若已配置则写入 `leaveRequests`
5. 后续管理员审批时写回原记录并追加 `approvalLogs`
6. 同时为员工生成审批结果消息

工资生成：

1. 读取目标月份请假、考勤、加班记录
2. 懒生成对应月份 `leaveBalances`（如缺失）
3. 计算带薪假使用、无薪假使用和结余
4. 基于账本结果生成工资项

## 9. 兼容策略

- 所有新增集合默认为空数组或空对象
- 历史数据没有 `notifications / leaveBalances / approvalSettings / approvalLogs` 时，页面必须正常工作
- 历史工资项若没有新拆分字段，前端按 `0` 展示
- 不做一次性全库迁移
- 不要求管理员先手动初始化所有月份账本

## 10. 测试策略

### 10.1 自动化测试

- 数据结构兼容测试
- 消息生成规则测试
- 假期账本生成与结转测试
- 工资联动测试
- 审批人未配置拦截测试
- 审批日志写入测试
- 页面层源码与行为测试
- 现有审批、工资、员工刷新等回归测试

### 10.2 核心场景

- 未配置审批人时，员工无法提交对应申请
- 配置审批人后，提交成功并进入待审批
- 审批后员工收到站内消息
- 公告与任务创建后，消息中心新增未读
- 月底结转规则正确影响次月余额
- 工资生成正确区分带薪假与无薪假
- 老数据缺少新字段时页面仍能打开

## 11. 上线策略

- 先在本地和预览数据验证
- 再部署到 `5176` 预览环境
- 用无痕窗口跑完整链路：
  - 员工提交请假
  - 管理员审批
  - 员工消息中心收到结果
  - 工资生成正确反映假期结果
- 预览通过后再切正式版 `5175`

## 12. 风险与取舍

### 风险

- 假期账本与工资逻辑联动，若设计不清容易产生历史月兼容问题
- 审批人未配置时若处理不当，会导致单据悬空
- 消息批量生成公告/任务提醒时，需控制性能和数据规模

### 取舍

- 首版只做站内消息，不做站外通知
- 首版只做单审批人配置，不做多级审批
- 首版保持任务可见范围与现状一致，不顺带重做任务分派
- 首版采用懒生成账本，不做全量历史迁移

## 13. 验收标准

- 员工端存在可用的消息中心，能看到公告、任务、审批结果消息
- 管理员可配置请假、加班、异常事项的审批人
- 未配置审批人时，员工提交会被明确拦截
- 假期余额可按月发放、消耗、结转
- 工资生成按带薪假/无薪假实际结果核算
- 审批动作具备统一日志可追溯
- 历史数据不因新结构而崩溃
