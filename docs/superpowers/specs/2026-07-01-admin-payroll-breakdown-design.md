# 后台工资编辑弹窗明细展示设计文档

## 1. 背景与目标

当前系统已经完成员工端工资页的考勤扣款明细展示，员工可以查看：

- 带薪假天数
- 无薪假天数
- 无薪假扣款
- 缺卡天数
- 缺卡扣款

后台 `工资核算` 页面目前仍停留在汇总编辑层，只能在编辑弹窗中查看和修改：

- 基础工资
- 加班费
- 扣款
- 实发工资

本次目标是在**后台工资页编辑弹窗**中，补充与员工端一致的工资拆分明细，帮助管理员在人工调整工资时，能同时看到扣款来源，但**不改变当前后台手动调薪逻辑**。

## 2. 范围

### 2.1 本次包含

- 在后台工资页编辑弹窗中新增只读的“考勤扣款明细”展示区。
- 展示以下字段：
  - 带薪假天数
  - 无薪假天数
  - 无薪假扣款
  - 缺卡天数
  - 缺卡扣款
- 保留现有 `基础工资 / 加班费 / 扣款 / 实发工资` 的人工编辑能力。
- 为后台页面补充中泰文案。
- 增加轻量回归测试，锁定后台弹窗中已接入这些字段与翻译键。

### 2.2 本次不包含

- 不把这些明细字段做成可编辑输入项。
- 不改动工资生成公式。
- 不新增新的工资数据字段。
- 不修改后台工资列表表格列结构。
- 不同步扩展 CSV 导出内容。
- 不重做后台工资弹窗的交互结构，仅在现有弹窗基础上补充只读信息区。

## 3. 方案选择

### 3.1 备选方案

本次讨论过三种方向：

- 方案 A：在后台工资编辑弹窗中新增只读明细区
- 方案 B：做成折叠面板，默认隐藏明细
- 方案 C：除了弹窗，还在后台工资列表里增加缩略明细

### 3.2 最终选择

采用**方案 A：在后台工资编辑弹窗中新增只读明细区**。

选择理由：

- 最符合当前用户诉求，范围聚焦。
- 不影响现有人工调整流程，学习成本最低。
- 数据字段已存在于 `PayrollItem`，接入成本低。
- 相比折叠面板，管理员打开弹窗后可直接看到扣款来源，更直观。
- 相比改表格列，更不容易让后台列表变拥挤。

## 4. 用户体验设计

### 4.1 管理员使用流程

管理员进入 `工资核算` 页面，选择月份并生成工资后，点击某一条工资记录的“编辑”。

弹窗中继续保留当前 4 个可编辑输入框：

- 基础工资
- 加班费
- 扣款
- 实发工资

在这些编辑项下方新增一块只读区域，展示本条工资记录的考勤扣款明细。管理员可以：

- 直接看到本月带薪假与无薪假的拆分情况
- 直接看到无薪假和缺卡造成的扣款金额
- 在保留总扣款人工调整能力的前提下，理解扣款来源

### 4.2 明细展示原则

- 明细区用于解释和辅助核对，不承担编辑职责。
- 明细内容与员工端工资页保持同口径，避免后台和员工端口径不一致。
- 历史工资数据如果没有这些字段，统一按 `0` 显示，不出现空白或报错。

## 5. 数据与状态设计

### 5.1 数据来源

本次展示直接读取 `PayrollItem` 已存在字段：

- `paidLeaveDays`
- `unpaidLeaveDays`
- `unpaidLeaveDeductionCents`
- `missingDays`
- `missingDeductionCents`

这些字段已经在工资生成流程中写入，不需要新增数据库结构或迁移逻辑。

### 5.2 历史数据兼容

旧工资记录可能创建于明细字段上线之前，因此展示层必须统一做兼容：

- 天数字段缺失时显示 `0`
- 金额字段缺失时显示 `0`

建议继续使用现有空值兜底方式：

- 数字：`value ?? 0`
- 金额：`formatCNYFromCents(value ?? 0)`

### 5.3 编辑边界

当前后台弹窗内的人工编辑能力保持不变：

- 可编辑：`baseSalaryCents`、`overtimePayCents`、`deductionsCents`、`netPayCents`
- 不可编辑：所有明细字段

这样可以保证：

- 管理员仍可按业务需要手动修正最终结果
- 不会出现“修改了总扣款，但明细字段是否要联动回算”的额外复杂度

## 6. 界面设计

### 6.1 布局结构

后台工资编辑弹窗维持当前纵向表单结构，在 4 个输入框下方增加一个独立信息区：

- 区块标题：考勤扣款明细
- 区块说明：用于提示该区域只展示本月带薪假、无薪假与缺卡对工资的影响
- 区块内容：5 个只读信息项

为了保持和员工端的视觉一致性，可沿用现有卡片样式思路，但不要求完全复制员工端布局。实现时重点是：

- 与当前后台弹窗风格一致
- 信息清晰、扫描成本低
- 不影响输入区可读性

### 6.2 字段展示

明细区展示以下内容：

- 带薪假天数
- 无薪假天数
- 无薪假扣款
- 缺卡天数
- 缺卡扣款

其中：

- 天数按整数或现有数值直接展示
- 金额使用现有货币格式化方法展示

### 6.3 为什么不直接复用员工端文案键

虽然员工端已有对应翻译键，但后台语义上属于管理员视角，建议为后台页面新增独立键：

- 便于后续分别调整员工端和后台表述
- 避免管理员页面直接依赖员工端命名
- 降低后续翻译维护耦合

## 7. 代码设计

### 7.1 涉及文件

- 修改 `d:\HR\src\pages\admin\Payroll.tsx`
  - 在编辑弹窗中新增只读明细区
- 修改 `d:\HR\src\i18n\translations.ts`
  - 增加后台工资明细展示相关的中泰翻译键
- 新增 `d:\HR\tests\adminPayrollBreakdown.test.ts`
  - 增加源码级轻量回归测试

### 7.2 组件职责

`src/pages/admin/Payroll.tsx` 继续承担后台工资页全部页面职责，本次不强制拆出新组件。

原因：

- 本次改动范围很小，属于现有弹窗的局部增强
- 逻辑来源简单，仅依赖 `openItem`
- 过早拆组件会增加额外文件和维护成本

如果实现过程中发现弹窗 JSX 明显膨胀，再考虑把明细区抽为一个局部只读展示组件，但这不是本次设计的硬要求。

### 7.3 数据流

数据流保持简单：

1. 管理员点击某条工资记录的“编辑”
2. 页面把该记录设为 `openItem`
3. 弹窗读取 `openItem` 中已有的工资明细字段
4. 明细区按只读方式渲染
5. 管理员保存时，仍只提交当前 4 个可编辑工资字段

本次不引入任何新的派生状态，也不在保存时写回明细字段。

## 8. 错误处理与边界情况

### 8.1 历史工资没有明细字段

处理方式：

- 用 `?? 0` 兜底显示

目标：

- 页面始终可展示
- 不因旧数据导致弹窗报错或布局异常

### 8.2 管理员手改总扣款后与明细区不一致

这是本次设计中允许存在的情况。

原因：

- 明细区反映的是系统生成工资时的原始拆分
- 总扣款和实发工资允许管理员因线下规则或额外处理进行人工修正

因此本次不做“总扣款变化后同步重算明细”的联动，也不把明细区解释为最终唯一依据。必要时可在后续版本中再考虑增加“系统计算值”与“人工调整值”的差异展示，但不纳入本次范围。

### 8.3 没有打开工资记录时

延续当前弹窗逻辑，不渲染弹窗正文。

## 9. 测试策略

### 9.1 轻量源码回归测试

新增一个与现有测试风格一致的轻量测试文件，验证：

- 后台工资页源码中已接入明细展示字段
- 后台工资页源码中已引用对应翻译键
- 翻译文件中存在这些中泰文案

这种测试方式适合本次需求，因为本次主要是页面显示增强，且当前项目已有类似测试模式。

### 9.2 基础回归验证

实现完成后至少验证：

- 后台工资页可以正常打开
- 点击编辑工资时弹窗正常出现
- 明细区正常显示
- 历史数据不会报错
- 修改 `基础工资 / 加班费 / 扣款 / 实发工资` 后仍可正常保存

### 9.3 构建验证

完成代码修改后需要至少跑：

- 相关测试
- `npm run build`

如果这次一并需要发布，再继续执行现有发布打包流程，但发布打包不作为本次设计本体的强制组成部分。

## 10. 验收标准

满足以下条件即可视为本次需求完成：

- 后台 `工资核算` 页面编辑弹窗中出现“考勤扣款明细”区
- 明细区显示：
  - 带薪假天数
  - 无薪假天数
  - 无薪假扣款
  - 缺卡天数
  - 缺卡扣款
- 这些字段为只读，不可编辑
- 现有 4 个工资输入项仍可编辑并正常保存
- 历史工资记录缺失字段时页面显示为 `0`，不会报错
- 中泰两套文案齐全
- 相关回归测试通过
