# HR 系统手机安装版（PWA）设计文档

## 1. 背景与目标

当前 HR 系统是基于 `React + Vite + TypeScript` 的单页网页应用，已经支持员工端与管理员端，并且已适配手机浏览器访问。用户希望在不改为原生 App、不中断现有局域网部署方式的前提下，把系统做成“手机桌面可安装”的应用形态，提升员工使用体验。

本次设计目标是把现有系统升级为标准 `PWA`（Progressive Web App）：

- 支持 Android 员工使用 `Chrome` 将系统安装到桌面。
- 支持 iPhone 员工使用 `Safari` “添加到主屏幕”。
- 安装后有独立图标，并以接近 App 的独立窗口方式打开。
- 保持当前局域网访问与便携发布版启动方式不变。
- 补充中文与泰文两套员工安装说明，便于实际推广使用。

## 2. 范围

### 2.1 本次包含

- 为现有 Vite 项目接入标准 PWA 能力。
- 增加 `manifest.webmanifest`、Service Worker、桌面图标、HTML 元信息。
- 优化手机端安装后的启动体验。
- 保留现有桌面端、员工端、管理员端业务逻辑。
- 在发布版中继续通过现有局域网共享脚本运行。
- 提供中文与泰文的手机安装说明。

### 2.2 本次不包含

- 不做 Android `APK` 安装包。
- 不做 iOS 原生 App 或 App Store / TestFlight 上架。
- 不做离线打卡、离线审批、离线工资计算。
- 不改动现有业务数据结构和审批/工资逻辑。
- 不把当前本地持久化方案替换成真实后端。

## 3. 用户体验设计

### 3.1 员工端目标体验

员工仍然先通过局域网地址访问系统，例如：

- `http://192.168.1.107:5177/`

首次打开后：

- Android：浏览器可出现安装提示，或员工可通过浏览器菜单选择“安装应用”/“添加到主屏幕”。
- iPhone：员工通过 `Safari` 菜单中的“添加到主屏幕”完成安装。

安装后：

- 手机桌面出现 HR 系统图标。
- 点击图标后以接近 App 的方式打开，不显示普通浏览器标签页形态。
- 打开后继续使用现有员工登录、打卡、请假、加班、工资查询流程。

### 3.2 管理端目标体验

管理员也可使用手机以同样方式安装，但本次重点仍是员工端体验提升。管理员相关界面不做额外移动端专属重构，只沿用现有响应式布局。

### 3.3 关键限制提示

安装后的 PWA 仍然依赖当前电脑主机运行：

- 主机上的黑色命令窗口不能关闭。
- 手机和主机必须处于同一局域网。
- 如果局域网 IP 或端口变化，员工需要重新打开最新地址。

## 4. 技术方案

### 4.1 方案选择

采用 `Vite PWA 插件` 接入，而不是手工维护全部 PWA 资源。

选择理由：

- 当前项目已基于 `Vite`，集成成本低。
- 可统一管理 `manifest`、缓存策略、Service Worker 注册。
- 后续如果还要增强手机端安装体验，可继续扩展。

### 4.2 实施方式

计划引入 `vite-plugin-pwa`，在现有构建流程中生成：

- `manifest.webmanifest`
- Service Worker
- 静态资源缓存配置

同时在应用入口自动注册 Service Worker，使部署后的系统可被浏览器识别为可安装应用。

### 4.3 缓存策略

本系统只做“安装能力 + 静态资源缓存”，不做完整离线业务。

缓存范围：

- `index.html`
- 打包后的 `js/css`
- 图标文件
- favicon 等基础静态资源

不对业务数据做离线同步承诺，原因是当前系统依赖运行中主机与浏览器本地数据，离线状态下继续操作反而会造成认知混乱。

### 4.4 对现有架构的影响

PWA 改造只涉及前端资源层与构建层：

- 不改 `dbStore` 的数据结构
- 不改登录逻辑
- 不改审批、工资、统计逻辑
- 不改现有便携版服务脚本的核心职责

因此这次改造风险较低，属于现有网页应用的增强，而不是重新做一套移动端。

## 5. 代码与文件设计

### 5.1 构建与配置

预计修改：

- `package.json`
  - 新增 PWA 依赖
- `vite.config.ts`
  - 接入 `vite-plugin-pwa`
  - 配置 `manifest`
  - 配置 Service Worker 注册策略

### 5.2 静态资源

预计新增：

- `public/manifest.webmanifest` 或通过插件内联生成
- `public/icons/` 目录
  - 至少提供 `192x192` 与 `512x512` 图标
  - 如有条件可增加 `180x180`（iPhone 友好）和 `maskable` 图标

### 5.3 HTML 与入口

预计涉及：

- `index.html`
  - 补充移动端 PWA 所需 meta
  - 包括主题色、Apple 移动端能力、图标声明等
- `src/main.tsx`
  - 注册 Service Worker

### 5.4 文档与发布说明

预计新增或更新：

- 发布版说明文档
- 员工手机安装说明（中文）
- 员工手机安装说明（泰文）

如需纳入发布包，可把说明文本放入发布版文件夹中，便于直接发给员工。

## 6. Manifest 设计

Manifest 需体现“企业员工考勤系统”的移动安装形态。

建议字段：

- `name`: 企业员工考勤系统
- `short_name`: HR考勤
- `start_url`: `/`
- `display`: `standalone`
- `background_color`: 与当前系统主题一致的深色背景
- `theme_color`: 与当前系统主色一致
- `lang`: `th`

图标策略：

- 一套统一品牌图标，避免依赖外部资源。
- 图标应兼顾深色/浅色桌面背景可见性。

## 7. 兼容性策略

### 7.1 Android

重点兼容：

- Chrome 最新版
- 常见国产 Android 浏览器中基于 Chromium 的安装能力

预期效果：

- 可安装到主屏幕
- 可独立窗口启动
- 支持现有拍照/上传逻辑（取决于浏览器权限）

### 7.2 iPhone

重点兼容：

- Safari

预期效果：

- 用户通过“添加到主屏幕”完成安装
- 启动后接近独立 App 体验

限制：

- iPhone 通常不会像 Android 一样主动弹出安装横幅
- 必须由用户手动从 Safari 菜单执行“添加到主屏幕”

## 8. 安装说明设计

### 8.1 中文版说明

需包含：

- 先连接公司同一 Wi-Fi
- 打开管理员提供的局域网地址
- Android / iPhone 的安装步骤
- 安装后如何从桌面打开
- 如果地址失效怎么办

### 8.2 泰文版说明

需包含与中文版等价的信息，便于直接发给泰国员工。

### 8.3 员工认知重点

说明中必须明确：

- 这是“可安装的网页 App”，不是应用商店下载安装包。
- 只要电脑主机在运行、网络一致，就能正常打开。
- 如果主机地址变化，需要重新用最新地址打开一次。

## 9. 发布与交付设计

### 9.1 对现有发布版的影响

现有发布方案仍然保留：

- `npm run build`
- `scripts/make-portable-release.mjs`
- 局域网共享启动脚本

PWA 能力会随着构建产物一并进入 `dist` 与发布版文件夹，不需要重新定义新的发布方式。

### 9.2 推荐交付物

最终交付物建议包含：

- 更新后的网页系统
- 便携发布版文件夹
- `.zip` 压缩包
- 中文手机安装说明
- 泰文手机安装说明

## 10. 风险与应对

### 10.1 IP/端口变化

风险：

- 员工第一次安装时的桌面图标绑定的是具体局域网地址。
- 如果主机 IP 或端口变化，桌面图标打开后会失败。

应对：

- 优先固定局域网端口。
- 如条件允许，后续可考虑固定主机 IP。
- 在员工说明中明确“地址变化后需重新打开最新地址”。

### 10.2 防火墙与局域网隔离

风险：

- 即使安装成功，如果主机防火墙阻止访问，员工仍无法打开。

应对：

- 继续使用现有自动放行防火墙脚本。
- 文档中明确优先使用自动放行版本启动脚本。

### 10.3 iPhone 安装认知差异

风险：

- 员工可能以为 iPhone 会自动弹安装提示，但通常不会。

应对：

- 在泰文与中文说明中单独写明 `Safari -> 分享 -> 添加到主屏幕`。

### 10.4 离线误解

风险：

- 用户可能误解为安装后可离线使用。

应对：

- 说明中明确：安装后更像 App，但仍需连接公司网络并依赖主机运行。

## 11. 验收标准

以下条件满足即视为本次设计目标达成：

- 在 Android Chrome 中可识别为可安装应用。
- 在 iPhone Safari 中可“添加到主屏幕”。
- 安装后桌面出现系统图标。
- 从桌面启动时以独立窗口打开。
- 现有员工端与管理员端业务不受影响。
- 便携发布版中包含完整 PWA 能力。
- 提供中文与泰文两份员工安装说明。

## 12. 测试策略

### 12.1 结构验证

- 构建成功
- 发布版可正常启动
- 浏览器可访问 Manifest
- Service Worker 正常注册

### 12.2 设备验证

- Android Chrome 安装流程验证
- iPhone Safari 添加到主屏幕流程验证
- 安装后重新打开登录页验证

### 12.3 回归验证

至少回归以下主流程：

- 员工登录
- 员工打卡页打开
- 请假/加班申请页打开
- 员工工资页打开
- 管理员登录
- 管理员审批页打开
