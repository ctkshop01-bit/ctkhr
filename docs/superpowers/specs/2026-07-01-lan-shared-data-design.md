# 局域网共享数据版设计文档

## 1. 背景与目标

当前系统虽然已经支持通过局域网地址访问，但本质上仍是：

- 主机只负责提供前端静态页面
- 账号与业务数据保存在各设备浏览器本地存储

当前关键问题是：

- 电脑端能登录的账号，手机端未必能登录
- 管理员在电脑端修改的数据，手机端不会同步
- 手机、电脑、PWA 安装版实际上各自维护独立数据

这与“局域网共享版”的真实预期不一致。

本次目标是把现有系统升级为**单主机、局域网共享、统一主数据版**：

- 主机保存唯一一份账号和业务数据
- 手机、电脑、PWA 都通过局域网访问这台主机上的同一份数据
- 保持现有 `.bat` 双击启动方式
- 保持当前 UI 和主要业务逻辑尽量不变
- 以**这台电脑当前浏览器中的现有数据**作为首次迁移的主数据来源

## 2. 范围

### 2.1 本次包含

- 新增主机端共享数据存储文件
- 新增主机端最小 API 服务
- 前端从本地持久化切换为请求主机共享数据
- 登录、员工管理、打卡、请假、加班、审批、工资等核心业务统一走主机数据
- 首次迁移时，以当前电脑现有浏览器数据导入主机主数据
- 保留现有局域网访问方式、PWA、便携版发布方式

### 2.2 本次不包含

- 不做云端同步
- 不做多主机数据同步
- 不做数据库服务器部署
- 不做复杂并发锁与事务系统
- 不做离线模式下的多端冲突合并
- 不把当前项目改造成前后端分离的复杂多服务架构

## 3. 现状分析

### 3.1 当前架构

目前局域网共享版由 [serve-dist.mjs](file:///d:/HR/scripts/serve-dist.mjs) 提供静态资源，主机启动后只是让其他设备访问同一个前端页面。

但真正的数据仍保存在浏览器本地：

- `am_db_v1`：业务数据
- `am_auth_v1`：登录态

其中：

- [dbStore.ts](file:///d:/HR/src/stores/dbStore.ts#L525-L544) 使用 `persist` 保存业务数据
- [authStore.ts](file:///d:/HR/src/stores/authStore.ts#L18-L43) 使用 `persist` 保存登录态

### 3.2 当前问题根因

因为业务数据是按浏览器本地存储保存的，所以：

- 电脑浏览器和手机浏览器各有自己的一份数据
- 手机看到的账号库不等于电脑看到的账号库
- 局域网访问地址共享了页面，但没有共享数据

### 3.3 当前启动链路

当前根目录启动脚本 [启动局域网共享版.bat](file:///d:/HR/启动局域网共享版.bat) 会：

1. 构建前端
2. 启动静态资源服务

发布版脚本 [make-portable-release.mjs](file:///d:/HR/scripts/make-portable-release.mjs) 也是围绕“静态网页 + 便携 Node”构建的。

## 4. 方案选择

### 4.1 备选方案

- 方案 A：主机 JSON 文件 + Node 最小 API
- 方案 B：SQLite + Node API
- 方案 C：继续依赖本地存储，只做同步补丁

### 4.2 最终选择

采用**方案 A：主机 JSON 文件 + Node 最小 API**。

选择理由：

- 与当前项目形态最兼容
- 可继续复用便携 Node 运行环境
- 不需要额外安装数据库
- 更适合现有 `.bat` 启动与发布版目录结构
- 能最快解决“多设备登录不一致”和“多端数据不一致”问题

### 4.3 为什么不选 SQLite

SQLite 在长期上更稳，但本次会显著增加：

- 打包复杂度
- 数据迁移复杂度
- 调试与维护成本

当前目标是先把“真正共享同一份数据”跑通，JSON 文件足以支撑这一步。

## 5. 总体设计

### 5.1 新架构

升级后的系统采用**单进程一体化服务**：

- Node 进程同时提供：
  - 前端静态资源
  - API 接口
  - 主机端 JSON 主数据读写

浏览器端不再把业务数据作为主存储，而是：

- 首次加载时从主机读取共享数据
- 业务操作通过 API 提交到主机
- 主机写入唯一主数据文件

### 5.2 主数据原则

主机上的共享数据文件是唯一主数据：

- 电脑浏览器不是主数据
- 手机浏览器不是主数据
- PWA 安装版也不是主数据

所有终端都以主机数据为准。

### 5.3 登录原则

登录校验不再依赖浏览器本地 `users` 数据，而是由主机 API 统一校验：

- 手机和电脑看到同一套账号
- 管理员修改账号/密码后全端立即生效

## 6. 数据设计

### 6.1 主机数据文件

新增共享数据目录，例如：

- `data/app-db.json`

文件内容结构沿用当前 `AppDb` 形态，尽量复用现有领域模型：

- `users`
- `clockEvents`
- `attendanceDaily`
- `leaveRequests`
- `overtimeRequests`
- `deductionRules`
- `payrollItems`
- `announcements`
- `announcementReads`
- `tasks`
- `taskAssignees`

这样可以减少业务重写成本。

### 6.2 登录态存储

浏览器端仍可保留轻量登录态存储，例如 token / userId / role，但它不再代表数据主库。

也就是说：

- 可以保留本地登录态持久化，提升刷新后的体验
- 但用户、工资、考勤等业务数据必须来自主机 API

### 6.3 写入策略

本次采用最小可行策略：

- 服务启动时将 `app-db.json` 读入内存
- 每次变更后整体写回 JSON 文件

这是最简单且足够稳定的单主机场景方案。

## 7. 首次迁移设计

### 7.1 关键限制

主机 Node 服务**不能直接读取浏览器 localStorage**。

因此“把这台电脑当前浏览器数据迁移到主机”不能靠服务端直接抓取，必须通过前端协助完成。

### 7.2 迁移方案

首次启动共享版时，按以下规则处理：

1. 如果主机 `data/app-db.json` 已存在，则直接使用，不再做导入
2. 如果主机 `data/app-db.json` 不存在，则系统进入“待初始化共享数据”状态
3. 在主机本机浏览器打开系统时：
   - 前端检查本机浏览器是否存在 `am_db_v1`
   - 如果存在，则提示“检测到当前电脑已有旧数据，是否导入为共享主数据”
4. 用户确认后：
   - 前端把浏览器里的旧数据提交到主机 API
   - 主机写入 `data/app-db.json`
5. 导入完成后：
   - 所有终端统一改读主机数据
   - 旧 localStorage 不再作为主数据来源

### 7.3 导入原则

本次导入以当前电脑现有数据为唯一主数据来源，不做多端合并。

原因：

- 这是用户明确选择的方案
- 多端合并会显著提升复杂度和出错风险

## 8. API 设计

### 8.1 设计原则

本次不追求完整 REST 细分，而采用**最小共享 API**，优先降低迁移成本。

### 8.2 建议接口

- `GET /api/health`
  - 健康检查
- `GET /api/bootstrap`
  - 返回服务状态、是否已初始化、当前用户可用的基础上下文
- `POST /api/bootstrap/import-local`
  - 首次导入当前电脑浏览器旧数据为主数据
- `POST /api/auth/login`
  - 统一登录
- `POST /api/auth/logout`
  - 退出登录
- `GET /api/db/snapshot`
  - 获取当前共享数据库快照
- `POST /api/db/command`
  - 提交业务命令，如打卡、编辑员工、审批、生成工资等

### 8.3 为什么用 command 模式

当前业务逻辑大量集中在 Zustand store action 中。为了减少一次性重构规模，可以把这些 action 逐步搬到服务端命令处理器，而不是一开始就拆成大量资源型 REST 接口。

这样能更快做到：

- 逻辑迁移最小化
- 前端界面改动最小化
- 后端逐步接管业务主逻辑

## 9. 前端改造设计

### 9.1 核心变化

前端需要从“本地数据库驱动”切换成“远端共享数据驱动”。

### 9.2 Store 策略

`dbStore` 不再把业务数据作为持久化主库，而改为：

- 保持内存态，用于驱动 UI
- 通过 API 拉取共享数据快照
- 通过 API 提交变更命令
- 收到成功响应后更新本地内存态

### 9.3 保留内容

可以继续保留：

- `authStore` 的轻量登录态持久化
- `i18nStore` 的语言偏好本地持久化
- 主题模式本地持久化

不需要共享到主机的数据，仍可保留在本地。

### 9.4 初始化流程

页面加载时：

1. 调用 `/api/bootstrap`
2. 判断主机共享库是否已初始化
3. 如果未初始化且当前设备是主机本机浏览器并存在旧 `am_db_v1`
   - 展示导入入口
4. 如果已初始化
   - 读取共享数据快照并驱动 UI

## 10. 服务端设计

### 10.1 新服务职责

新的主机服务需要承担：

- 静态资源服务
- API 路由分发
- JSON 数据加载与保存
- 业务命令执行
- 首次导入旧数据

### 10.2 代码组织建议

建议新增以下职责模块：

- `server/appDbFile.ts`
  - 读写 `app-db.json`
- `server/appDbMemory.ts`
  - 进程内共享状态
- `server/commands/*.ts`
  - 登录、员工编辑、打卡、审批、工资等命令处理
- `server/httpServer.ts`
  - 路由与响应

当前 `serve-dist.mjs` 可演进为统一服务入口，而不是纯静态文件服务器。

### 10.3 安全边界

本次仍是局域网内部使用，因此安全目标为“够用且稳”：

- 不做复杂 token 系统
- 不做外网暴露假设
- 但至少要保证基础登录校验和角色边界不回退

## 11. 启动与发布设计

### 11.1 启动方式

继续保留现有双击启动体验：

- `启动局域网共享版.bat`
- `启动局域网共享版_自动放行防火墙.bat`

但其实际启动目标从“静态前端服务”升级为“共享数据服务 + 前端资源服务”。

### 11.2 发布包结构

发布版中新增共享数据目录，例如：

- `data/app-db.json`

以及新的统一服务脚本。

### 11.3 备份

因为主机数据成为唯一主数据，后续备份重点应转移到：

- 发布版目录中的共享数据文件

未来可再增加一键备份脚本，但不纳入本次范围。

## 12. 错误处理与边界情况

### 12.1 主机未初始化

如果主机共享库还没建立：

- 非主机设备不能直接进入正常业务页
- 需提示“请先在主机电脑完成首次数据导入”

### 12.2 主机初始化后旧 localStorage 仍存在

这是允许存在的，但它不再是主数据。

前端应明确：

- 已初始化后统一读取主机数据
- 不再继续把旧 localStorage 合并回主机

### 12.3 主机数据文件损坏

至少要做到：

- 读取失败时给出明确错误
- 不要悄悄回退到种子数据覆盖掉主数据

### 12.4 多设备同时操作

本次采用单进程内存 + 顺序写文件方式，能覆盖轻量局域网并发场景。

不保证复杂高并发一致性，但对当前 HR 使用规模足够。

## 13. 测试策略

### 13.1 服务端测试

重点覆盖：

- JSON 读写
- 首次初始化逻辑
- 从旧 local 数据导入共享库
- 登录校验
- 关键命令处理

### 13.2 前端测试

重点覆盖：

- 初始化共享模式后的页面加载
- 主机未初始化时的提示逻辑
- 导入完成后的切换逻辑
- 登录改为 API 后的行为

### 13.3 集成验证

至少验证以下真实流程：

1. 主机首次启动共享版
2. 用当前电脑旧数据导入主机库
3. 电脑端正常登录
4. 手机端用同一账号密码正常登录
5. 管理员改密码后，手机端使用新密码登录成功
6. 员工打卡、请假、审批、工资生成跨设备同步可见

## 14. 验收标准

满足以下条件即可认为本次目标达成：

- 手机和电脑共用同一套账号密码
- 主机保存唯一主数据
- 管理员修改账号/密码后全端同步生效
- 考勤、请假、加班、审批、工资在不同设备间同步一致
- 首次升级时可把这台电脑当前旧数据导入为主数据
- `.bat` 启动和局域网访问方式保留
- 发布版、`.zip`、`_PWA` 目录仍可正常交付使用

## 15. 后续可扩展项

本次不做，但后续可以独立追加：

- JSON 自动备份与历史版本回滚
- SQLite 存储升级
- 更细粒度 API 拆分
- 局域网内实时推送刷新
- 多主机容灾与云同步
