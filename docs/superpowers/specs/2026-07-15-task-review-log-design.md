# 任务审核日志设计文档

日期：2026-07-15

## 1. 背景

第一期 KPI / 绩效主链路已经上线了任务提交、管理员审核、绩效事件生成和员工绩效展示能力，当前管理员可以在审批页处理任务提交，员工也能看到任务状态变化。

但现阶段仍有一个明显缺口：系统只保留任务的当前状态，没有完整保留任务审核过程。进入真实使用后，管理员和员工很快会遇到以下问题：

- 谁审核了这条任务
- 是什么时候审核的
- 任务为什么被退回
- 同一任务是否被多次退回
- 员工是否已经重新提交过

如果这些问题只能靠当前任务字段推断，后续一旦出现争议、复核、人工调整绩效或管理追责，系统就缺乏足够的过程证据。

因此，第二期优先补齐“任务审核日志”能力，目标不是重新设计整个审批系统，而是在现有第一期结构上最小增量地补上任务审核过程留痕。

## 2. 目标

### 2.1 业务目标

- 让管理员能追溯任务审核全过程
- 让员工能看到最近一次处理结果和退回原因
- 为后续绩效申诉、人工调整绩效、责任追溯提供基础证据
- 降低“任务被退回但说不清原因”带来的管理争议

### 2.2 工程目标

- 保持与现有 `dbStore -> sharedApi -> app-db-commands.mjs` 模式一致
- 不推翻第一期任务状态与绩效事件模型
- 让任务当前状态和审核过程历史分层存储、职责清晰
- 采用最小可上线范围，先解决追溯问题，再考虑导出和全局查询

## 3. 范围

### 3.1 本次包含

- 新增独立的任务审核日志数据结构 `taskReviewLogs`
- 在任务提交、审核通过、审核退回时自动写入日志
- 管理员在审批页任务审核弹窗中查看某条任务的审核记录
- 员工在任务卡片或任务详情中查看最近一次处理结果
- 保留退回原因与审核操作人信息

### 3.2 本次不包含

- 审核日志导出
- 全局日志中心
- 高级筛选和多维查询
- 与现有 `approvalLogs` 完全合并
- 员工侧完整审核时间线

## 4. 方案选择

### 4.1 备选方案

- 方案 A：新增独立 `taskReviewLogs`，任务状态继续保存在 `task`，审核过程保存在日志集合
- 方案 B：继续扩展 `task` 字段，只保留最后一次审核信息，不建历史集合
- 方案 C：强行复用现有 `approvalLogs`，把任务审核塞进通用审批日志模型

### 4.2 最终选择

采用 **方案 A：独立任务审核日志集合 + 审批弹窗内嵌展示 + 员工侧只看最近处理结果**。

### 4.3 选择理由

- 最符合当前代码现状，改动面最小
- 能完整保留过程历史，而不仅仅是最终状态
- 不污染现有 `approvalLogs` 的通用审批语义
- 后续可自然扩展到绩效申诉、人工调整绩效、责任追溯等功能

## 5. 总体方案

本次采用“当前状态 + 历史日志”双层模型：

- `tasks` 继续保存任务当前状态，用于页面展示和业务判断
- `taskReviewLogs` 保存所有审核动作，用于追溯和只读展示
- 任务动作仍然沿用现有服务端命令入口，不新增独立的日志写入命令
- 管理员审批弹窗中直接展示当前任务的审核历史
- 员工侧先只展示最后一次处理结果，避免第二期范围膨胀

该方案的核心原则是：

- `task` 负责“现在是什么状态”
- `taskReviewLogs` 负责“过程发生过什么”
- `performanceEvents` 继续负责“这件事如何计分”

三者职责分开，不互相混淆。

## 6. 数据模型

### 6.1 新增 taskReviewLogs

在 `AppDb` 中新增：

- `taskReviewLogs: TaskReviewLogItem[]`

### 6.2 TaskReviewLogItem 字段建议

- `id`
- `taskId`
- `userId`
- `action`
- `fromStatus`
- `toStatus`
- `operatorUserId`
- `operatorRole`
- `reason`
- `createdAtISO`

### 6.3 字段说明

- `taskId`：关联的任务 ID
- `userId`：任务归属员工 ID
- `action`：本次动作类型，首版支持：
  - `submit`
  - `confirm`
  - `return`
- `fromStatus`：变更前任务状态
- `toStatus`：变更后任务状态
- `operatorUserId`：本次操作人 ID
- `operatorRole`：操作人角色，便于后续区分主管和管理员
- `reason`：提交备注或退回原因；`confirm` 可为空
- `createdAtISO`：日志创建时间

### 6.4 与现有模型的关系

- `tasks` 不删除现有 `submittedAtISO`、`confirmedAtISO`、`lastReturnReason` 等字段
- 这些字段继续承担“当前任务快捷展示”的作用
- `taskReviewLogs` 负责提供完整历史，不要求前端从任务字段反推全过程

## 7. 交互设计

### 7.1 管理员端

管理员入口继续放在现有审批页任务审核弹窗中：

- 文件位置：`src/pages/admin/Approvals.tsx`
- 不新增独立日志页面

弹窗中新增一个 `审核记录` 区块，放在任务描述和审核备注下方，按时间倒序展示。

每条记录展示：

- 动作：`已提交 / 已通过 / 已退回`
- 操作人
- 操作时间
- 原因或备注

目标是让管理员在处理任务时，不必跳走页面，就能看到这条任务此前发生过什么。

### 7.2 员工端

员工端首版不展示完整审核时间线，只展示与自己最相关的最后处理结果：

- 最近处理状态
- 最近处理时间
- 退回原因

展示位置可放在：

- `Tasks.tsx` 的任务卡片扩展区
- 或任务详情抽屉/弹窗

首版优先级更高的是“员工看得懂为什么被退回”，不是“员工查看完整操作流水”。

## 8. 服务端命令设计

### 8.1 设计原则

不新增独立的“写日志命令”，而是在现有任务动作命令中内聚完成日志写入。

这样可以保证：

- 任务状态和日志写入在同一个事务性步骤中完成
- 前端仍沿用现有命令入口，不需要新增额外调用
- 数据一致性由服务端统一保障

### 8.2 命令映射

- `submitTaskCompletion`
  - 更新任务状态为 `submitted`
  - 同时写入一条 `submit` 日志

- `reviewTaskCompletion(confirm)`
  - 更新任务状态为 `confirmed`
  - 同时写入一条 `confirm` 日志

- `reviewTaskCompletion(return)`
  - 更新任务状态为 `returned`
  - 同时写入一条 `return` 日志

### 8.3 一致性要求

每次任务动作必须同时成功完成以下两件事：

- 更新 `task` 当前状态及相关字段
- 追加一条 `taskReviewLogs`

如果任一环节失败，则整次操作失败，不允许出现：

- 任务状态已变更，但没有对应日志
- 已生成日志，但任务状态未成功变更

## 9. 前端读取策略

### 9.1 管理员端读取

管理员审批弹窗按 `taskId` 过滤 `taskReviewLogs`，取结果后按时间倒序展示。

展示内容重点是：

- 最近一次是谁处理
- 处理结果是什么
- 是否有多次退回
- 退回原因是什么

### 9.2 员工端读取

员工任务页不直接展示全部日志，只取最近一条 `confirm` 或 `return` 类型日志，用于展示：

- 最近结果
- 最近处理时间
- 最近退回原因

这样能满足最基础的可解释性，同时保持第二期实现成本可控。

## 10. 错误处理

### 10.1 服务端

- 如果任务不存在，命令直接失败，不写日志
- 如果 `return` 动作没有原因，首版建议仍允许为空，但前端继续保持“退回时建议填写原因”
- 如果日志生成失败，整个任务动作失败，保持数据一致

### 10.2 前端

- 审核记录为空时，管理员弹窗显示空状态提示
- 员工侧没有最近处理记录时，不报错，显示默认状态文案
- 日志加载失败时，沿用现有共享快照错误处理方式

## 11. 兼容策略

- 历史任务没有任何审核日志时，系统仍应正常打开
- 老任务只依赖当前 `task` 字段继续展示，不强制要求补齐历史日志
- 新增 `taskReviewLogs` 缺失时，默认视为空数组
- 不影响第一期已存在的 `approvalLogs`、`performanceEvents`、`performanceMonthlySummaries`

## 12. 测试策略

### 12.1 自动化测试

- 提交任务后写入 `submit` 日志
- 审核通过后写入 `confirm` 日志
- 审核退回后写入 `return` 日志，并保留原因
- 日志的 `fromStatus` / `toStatus` 正确
- 审批页能读取任务日志并展示对应文案

### 12.2 核心场景

- 员工提交任务，管理员打开审批弹窗可看到 `submit` 日志
- 管理员退回任务后，审批弹窗显示退回历史，员工侧可看到最新退回原因
- 员工再次提交后，日志中形成完整的 `submit -> return -> submit` 过程
- 管理员最终确认后，日志中形成完整闭环

## 13. 风险与取舍

### 风险

- 如果后续又把任务审核并入 `approvalLogs`，会产生模型重叠
- 如果员工端过早暴露完整日志，会带来更多解释和展示复杂度
- 如果任务动作和日志写入不保持原子性，会导致状态与历史不一致

### 取舍

- 第二期先做“任务审核留痕”，不做全局日志中心
- 员工端先做“最近结果可见”，不做完整历史时间线
- 不把日志直接并入绩效计分，避免职责混淆

## 14. 验收标准

- 任务提交时自动生成 `submit` 日志
- 审核通过时自动生成 `confirm` 日志
- 审核退回时自动生成 `return` 日志，并能保留原因
- 管理员审批弹窗能看到某条任务的审核历史
- 员工端能看到最近处理结果和退回原因
- 历史数据缺失日志时，页面不会报错

## 15. 实施建议

建议按以下顺序实施：

1. 扩展 `domain.ts` 与 `seedDb.ts`，加入 `TaskReviewLogItem` 和 `taskReviewLogs`
2. 扩展 `app-db-commands.mjs`，在现有任务动作中内聚写日志
3. 扩展 `dbStore.ts`，保持命令入口不变或最小调整
4. 修改 `Approvals.tsx`，在任务审批弹窗展示审核记录
5. 修改员工任务页，展示最近处理结果与退回原因
6. 补齐自动化测试与回归验证

## 16. 结论

任务审核日志是第二期最合适优先补齐的能力，因为它解决的是当前最真实、最容易立刻遇到的管理问题：任务审核过程不可追溯。

采用独立 `taskReviewLogs` 的方案，能够在不推翻第一期结构的前提下，以最小改动补齐过程留痕，并为后续绩效申诉、人工调整绩效、审核责任追溯打下稳定基础。
